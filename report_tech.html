1. Embedding (임베딩)

텍스트를 고차원 벡터(숫자 배열)로 변환하는 기술. 의미가 유사한 텍스트는 벡터 공간에서 가까운 위치에 매핑된다.
"센서 설치 방법"
→
Embedding Model
text-embedding-3-small
→
[0.23, -0.87, 0.45, ... ] (1536차원)

유사 문장 → 유사 벡터 → 코사인 유사도(Cosine Similarity)로 의미적 거리 계산 가능.

2. Vector DB (벡터 데이터베이스)

벡터를 저장하고 유사도 기반으로 검색(ANN: Approximate Nearest Neighbor)하는 데이터베이스. 키워드 매칭이 아닌 의미적 유사도로 검색한다.

키워드 검색: "측정 범위" → 해당 단어 포함 문서만

벡터 검색: "측정 범위" → "센서 감지 영역", "측정 가능 거리" 등 의미적으로 유사한 문서 전부 반환

본 시스템: ChromaDB (HNSW 인덱스, 코사인 유사도)

저장: 제품 문서 12개 → 500자 청크 분할 → 138개 벡터 저장

검색: 질문 벡터와 가장 가까운 청크 Top-5 반환

3. RAG (Retrieval-Augmented Generation)

LLM이 답변 생성 시, 외부 지식을 검색(Retrieval)하여 컨텍스트로 주입한 뒤 생성(Generation)하는 아키텍처. LLM의 환각(Hallucination)을 줄이고 도메인 특화 답변을 가능하게 한다.
Query
→
Embedding
→
Vector DB
Top-K 검색
→
LLM
Context + Query → 답변

Naive LLM: 학습 데이터(파라메트릭 지식)에만 의존 → 도메인 지식 부족, 환각 발생

RAG: 비파라메트릭 지식(외부 문서)을 프롬프트에 주입 → 근거 기반 답변, 문서 업데이트 즉시 반영

본 시스템 구성:

Embedding: OpenAI text-embedding-3-small (1536d)

Vector DB: ChromaDB (코사인, Top-5)

LLM: GPT-5.4-mini (temperature 0.3, stream)

Context window: 검색 문서 5개 + 대화 히스토리 20개
